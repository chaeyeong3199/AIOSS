# 사용자 여정 시나리오 (User Journey Scenario) — LightGauss Optimizer

## 1. 개요
LightGauss Optimizer는 GPS-Gaussian 모델을 경량화하고 가속화하는 도구로, 그래픽 엔진 개발자, VR/AR 애플리케이션 개발팀, 연구기관 시각화 연구자를 대상으로 한다. 이 시나리오는 사용자가 제품을 처음부터 최적화된 모델을 배포하기까지의 여정을 단계별로 설명한다. 각 단계에서 사용자의 목표, 행동, 감정, 터치포인트(도구/인터페이스)를 포함한다.

---

## 2. 주요 사용자 페르소나
- **페르소나 1: 그래픽 엔진 개발자 (Alex)**  
  배경: 대형 게임 엔진 팀에서 일하는 개발자. VR/AR 앱의 실시간 렌더링 성능을 최적화해야 함.  
  목표: 메모리 사용을 줄이고 프레임율을 높여 사용자 경험 향상.  
  동기: 클라이언트 요구사항 충족과 프로젝트 데드라인 준수.

  배경: 스타트업에서 모바일 VR 앱 개발. 제한된 하드웨어에서 고품질 렌더링 필요.  
  목표: 모델 경량화로 배터리 수명 연장과 성능 향상.  
  동기: 시장 경쟁력 확보와 사용자 만족도 증대.

- **페르소나 3: 연구기관 시각화 연구자 (Taylor)**  
  배경: 대학 연구실에서 3D 시각화 연구. 대규모 데이터셋 처리.  
  목표: 효율적인 모델로 실험 시간 단축과 정확도 유지.  
  동기: 학술 논문 발표와 연구 자금 확보.

## 3. 사용자 여정 단계

### 단계 1: 모델 준비 및 파이프라인 적용 (Awareness & Consideration)
**목표:** 원본 GPS-Gaussian 모델을 가져와 최적화 파이프라인을 실행하여 경량화된 모델 생성.  
**행동:**  
- GitHub 또는 공식 사이트에서 LightGauss Optimizer 다운로드 및 설치.  
- 터미널에서 CLI 명령어로 원본 모델 파일 로딩 (예: `lightgauss optimize --input model.ply --output optimized.ply`).  
- 프루닝 비율과 샘플링 파라미터 설정 (기본값 사용 또는 커스터마이징).  
- 파이프라인 실행 (약 10분 소요).  
**감정:** 기대감 (새 도구 시도), 약간의 불안 (처음 사용).  
**터치포인트:** CLI 인터페이스, 로그 출력, 오류 메시지.  
**성과:** 메모리 30% 이상 감소한 최적화 모델 생성.  
**페인포인트:** 파라미터 설정이 복잡할 수 있음; 가이드 문서 필요.

### 단계 2: 테스트 및 측정 (Trial & Evaluation)
**목표:** 최적화된 모델을 테스트 환경에 배포하여 성능 측정.  
**행동:**  
- VR/AR 엔진에 모델 통합 (예: Unity 또는 Unreal Engine 플러그인 사용).  
- 테스트 디바이스(VR 헤드셋, 모바일)에서 렌더링 실행.  
- 벤치마크 도구로 메모리 사용량, 프레임율 측정 (예: `lightgauss benchmark --model optimized.ply --device vr`).  
- 결과 CSV/그래프 저장.  
**감정:** 만족 (성능 향상 확인), 호기심 (더 세부 조정 가능성).  
**터치포인트:** 엔진 인터페이스, 벤치마크 스위트, 보고서 출력.  
**성과:** 프레임율 2배 향상, 메모리 50% 감소 확인.  
**페인포인트:** 디바이스별 호환성 문제; 추가 설정 필요.

### 단계 3: 품질 비교 및 반복 최적화 (Purchase & Adoption)
**목표:** 시각적 품질을 검증하고 파라미터 조정으로 반복 최적화.  
**행동:**  
- PSNR/SSIM 도구로 원본 vs. 최적화 모델 비교 (예: `lightgauss compare --original model.ply --optimized optimized.ply`).  
- 품질 보고서 확인 (PSNR ≥30dB, SSIM ≥0.95 유지).  
- 불충분 시 파라미터 재조정 (프루닝 임계값 낮추기, 샘플링 가중치 변경).  
- 반복 실행 및 재평가.  
- 만족 시 모델 버전 관리 (Git LFS 연동) 및 배포.  
**감정:** 성취감 (목표 달성), 안도 (품질 유지).  
**터치포인트:** 품질 측정 도구, 파라미터 설정 인터페이스, 버전 관리 시스템.  
**성과:** KPI 달성 (메모리 ≥50% 감소, 프레임율 ≥2배, 품질 유지).  
**페인포인트:** 반복 과정이 시간 소요; 자동화된 튜닝 기능 필요.

### 단계 4: 배포 및 모니터링 (Retention & Advocacy)
**목표:** 최적화된 모델을 프로덕션에 배포하고 성능 모니터링.  
**행동:**  
- 모델을 앱/엔진에 통합하여 출시.  
- 사용자 피드백 수집 (프레임율, 메모리 사용).  
- 모니터링 도구로 실시간 성능 추적.  
- 필요 시 업데이트 (새 데이터나 기법 적용).  
**감정:** 자신감 (성공적 배포), 충성심 (제품 만족).  
**터치포인트:** 배포 플랫폼, 모니터링 대시보드, 사용자 설문.  
**성과:** 안정적 운영, 사용자 만족도 향상.  
**페인포인트:** 장기 모니터링 데이터 부족; 통합 가이드 개선 필요.

---

## 4. 전체 여정 요약
- **총 기간:** 1-2주 (단일 프로젝트 기준).  
- **주요 성공 요인:** 직관적 CLI, 정확한 측정 도구, 유연한 파라미터 조정.  
- **개선 제안:** GUI 인터페이스 추가, 자동 튜닝 기능, 더 많은 디바이스 지원.  
- **리스크:** 품질 저하 시 사용자 이탈; 대응으로 후처리 필터 강화.
